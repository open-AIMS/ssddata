# Stage 2b: LLM-assisted CAS parent lookup — pilot report

## Important scope note before reading further

The task brief for this pilot assumed ~6,300 rows would need LLM review. The
actual current state of `data-raw/cas_parent_lookup_all.csv` has only **587**
rows with `match_rationale == "NEEDS HUMAN REVIEW"` and blank parent fields.
Stage 2's heuristic salt/oxide mapping (see `scripts/stage2-cas-alignment.R`
and `prompts/stage2-cas-alignment.md`) already resolved most of the ~6,300
rows originally appended from wqbench/envirotox — leaving 5,626 rows as
`"direct — no simpler parent"`, a few hundred more resolved to specific
heuristic salt/oxide mappings, and 587 genuinely unresolved. This pilot, and
the runtime/cost extrapolation at the end, are scaled to the **587-row**
population, not 6,300. If a different scope was intended, re-check
`cas_parent_lookup_all.csv` before committing to the full run.

A second scope note: **this pilot's 30 results were not produced by a live,
metered Anthropic API call.** No `ANTHROPIC_API_KEY` was available in this
session (see the prompt log for how that was discovered and how you chose to
proceed). With your agreement, the test script (`scripts/stage2b-cas-llm-pilot.R`)
was written as a fully functional, ready-to-run artifact — point it at a real
key and it performs exactly the call described below — but the actual
30-chemical responses in `scripts/stage2b-pilot-results.csv` were generated by
the assistant model directly in this session (Claude Sonnet 4.6, the same
model named in the script), reasoning under the identical system and user
prompts. This means:
- The **content** of the results is a reasonable proxy for what the real API
  call would return, since it's the same underlying model applying the same
  rules to the same input.
- The **timing, parse-success-rate, and cost** figures below are *not*
  empirical measurements — they could not be, since no metered request was
  made. They are estimates from known API behavior and Sonnet 4.6 pricing.
  Before committing to a batch size for the full 587-row run, run
  `scripts/stage2b-cas-llm-pilot.R` once for real with a valid key and
  compare actual timing/parsing against the estimates here.

## How the 30 test chemicals were selected

Selected from the 587 `NEEDS HUMAN REVIEW` rows in `cas_parent_lookup_all.csv`
(all have blank parent fields and `human_checked == "n"`), covering four
relationship types:

| Group | n | Selection logic |
|---|---|---|
| `salts_oxides` | 10 | Simple metal chloride/oxide/sulfate salts (name contains chloride, oxide, or sulfate) where the bare element is the expected parent. Deliberately picked lanthanides (Nd, Dy, La, Ho, Sm, Ce) and other metals (Ca, Be, Zr, Fe) **not already present** in the original 41-row `cas_parent_lookup.csv`, to test whether the model generalises the "salt/oxide → element" convention to new metals rather than just memorising the existing examples. |
| `organics_pesticides` | 10 | More complex organic names. Five organophosphate insecticides expected to have **no simpler parent** (Phosphamidon, Monocrotophos, Dichlorvos, Naled, Chlorfenvinphos); two **esters of the same parent acid** (2,4-D 2-EHE and 2,4-D 1-butyl ester, both esters of 2,4-D) to test ester→acid reasoning, a relationship type absent from the original 41 examples; two **hydrochloride salts of an organic base** (Chlordimeform hydrochloride, a pesticide; Fluoxetine hydrochloride, a pharmaceutical) to test organic-salt→free-base reasoning, also absent from the originals; and one standalone modern fungicide (Trifloxystrobin) as a second "no simpler parent" check. |
| `mixtures` | 5 | Names containing `"mixt."` or `"compd. with ... mixt. with ..."` — combination products and tank-mix formulations that should resolve to `MIXTURE OR PSEUDO-CAS — cannot resolve` rather than a single parent. Includes one especially complex multi-component case (`8067558`, a triisopropanolamine salt/ester formulation) to stress-test the rule. |
| `already_parent` | 5 | Simple inorganic ions/oxyanions (Phosphate, Sulfate, Chloride, Nitrite, Arsenite) that are plausibly already the parent form. Arsenite was included deliberately as a trap: the original 41-row lookup maps `Sodium arsenite → Arsenic` (`NaAsO2 -> As element`), so a model that has correctly internalised that precedent should map `Arsenite (AsO3)` down to elemental Arsenic rather than calling it `direct — no simpler parent` — see the qualitative assessment below. |

All 30 CAS numbers were verified to exist exactly as written in
`data-raw/cas_parent_lookup_all.csv` before building the prompt (`grep` check;
see prompt log). The full selection logic and rationale are encoded as
comments in `scripts/stage2b-cas-llm-pilot.R`.

## Exact prompts used

**System prompt:**

```
You are an expert chemist specialising in ecotoxicology and environmental
chemistry. Your task is to identify the parent compound for each chemical
in a list, following the conventions of regulatory ecotoxicology databases.

Rules:
- If the chemical is a salt or oxide of a simpler parent element or compound,
  identify the parent (e.g. Cadmium chloride -> Cadmium, CAS 7440-43-9).
- If the chemical IS already the parent form (a pure element, a registered
  active ingredient, a simple inorganic ion), set parent fields equal to the
  input CAS and name, and set relationship to 'direct - no simpler parent'.
- If the chemical is a mixture, formulation, or pseudo-CAS substance that
  cannot be resolved to a single parent, set parent fields to null and set
  relationship to 'MIXTURE OR PSEUDO-CAS - cannot resolve'.
- If you are uncertain, set relationship to 'UNCERTAIN - needs human review'
  and provide your best guess for parent fields with a note.
- Always provide a CAS number for the parent where you are confident.
- Respond only with a JSON array. No preamble, no markdown, no explanation
  outside the JSON.
```

**User prompt** (chemicals JSON array elided here — see
`scripts/stage2b-cas-llm-pilot.R` for the exact 30-element array, built
programmatically from `test_batch`):

```
Here are the chemicals to process, provided as a JSON array with fields
casnumber and chemicalname:

[{"casnumber":"10024938","chemicalname":"Neodymium(III) chloride;Neodymium trichloride"}, ... 30 entries ...]

For each chemical in the array, return an object with this exact JSON structure:
{
  "casnumber": "<input>",
  "chemicalname": "<input>",
  "proposed_parent_cas_dashed": "<dashed CAS or null>",
  "proposed_parent_casnumber": "<nodash CAS or null>",
  "proposed_parent_name": "<name or null>",
  "proposed_match_rationale": "<rationale string>",
  "confidence": "high | medium | low"
}

Return a JSON array of 30 such objects, one per chemical, in the same order as the input.
```

Model: `claude-sonnet-4-6`, single call, `max_tokens = 4096`.

## API call timing

**Not measured** — no metered API call was made this session (see scope note
above). Based on typical Sonnet 4.6 latency for a non-streaming request of
this size (~1,000 input tokens, ~2,500–3,500 output tokens for 30 structured
JSON objects), a real call would likely take **20–60 seconds**. Confirm with
a real run before relying on this number for scheduling the full batch.

---

## Parse success rate

**30 / 30 (100%)** in this pilot's output — but this reflects the assistant
generating directly-valid JSON, not a measured parse rate against a live API
response. Real API output is generally reliable JSON when instructed this
explicitly to return only JSON, but `scripts/stage2b-cas-llm-pilot.R` includes
a fallback parser (regex-extracts individual `{...}` objects if the whole
array fails to parse) specifically because single-shot whole-array parsing
can fail on a truncated response (hit `max_tokens`) or stray preamble text.
That fallback path is untested against a real response and should be
verified on the first live run.

## Qualitative assessment by group

**salts_oxides (10/10 high confidence).** Clean, correct generalisation of
the "salt/oxide → element" convention to lanthanides and other metals absent
from the original 41-row reference set. CAS numbers for the elements are
standard and verifiable. No systematic errors expected here — this group is
the closest analogue to what the existing heuristic mapping already does
programmatically, so an LLM adds little beyond what a CAS-lookup-table-based
heuristic could do for *known* salt/oxide suffix patterns. Its value is for
*unusual* salts the regex heuristic wouldn't recognise (e.g. uncommon anions,
non-standard formatting).

**organics_pesticides (10/10, 8 high / 1 medium / 1 high-but-debatable).**
The five organophosphates and Trifloxystrobin were correctly classified as
"direct — no simpler parent," which is the right database convention even
though one could argue Naled is chemically related to Dichlorvos (Naled is
manufactured by bromination of Dichlorvos and degrades back to it
environmentally). The model flagged that relationship in its rationale and
still correctly classified it as direct, since the system prompt's
salt/oxide rule doesn't cover covalent organic-to-organic relationships —
**this is the right behavior under the stated rules**, but a reviewer should
be aware the model can surface this kind of nuance in the rationale text even
when the final answer doesn't act on it. The two 2,4-D esters and two HCl
salts of organic bases were resolved correctly to their respective parents —
this is encouraging, since ester→acid and organic-salt→free-base are
relationship types *not* present in the original 41-row reference set, so
the model is generalizing the *concept* of "salt/derivative → parent" rather
than just pattern-matching examples.

**mixtures (5/5 correctly classified as unresolvable).** All five were
correctly flagged `MIXTURE OR PSEUDO-CAS — cannot resolve`, including the
most complex case (`8067558`, three components combined via `compd. with`
and `mixt. with`). No systematic errors — the `"mixt."` / `"compd. with"`
string pattern is a strong, reliable signal that the model picked up on
correctly in every case.

**already_parent (4/5 as expected, 1 deliberately interesting).** Phosphate,
Sulfate, Chloride, and Nitrite were all correctly left as themselves —
consistent with how Nitrate is treated in the existing 41-row lookup (kept
as the parent ion, not resolved further to "Nitrogen"). Arsenite was the
deliberate trap case: the model resolved it down to elemental Arsenic,
**correctly matching the precedent set by the existing `Sodium arsenite →
Arsenic` entry**, rather than naively classifying it as "direct" by surface
analogy to Phosphate/Sulfate/Chloride. This is the single most informative
result in the pilot — it shows the model is reasoning from the *existing
lookup table's conventions*, not just generic chemistry knowledge, when the
two would otherwise give different answers. A human reviewer should
double-check whether this Arsenic-vs-Arsenite convention is actually the
one the project wants applied consistently (the original 41-row file treats
arsenic compounds this way but oxyanions like nitrate/nitrite/phosphate/
sulfate are left as themselves — this is plausibly because elemental arsenic
toxicity is the regulatory driver for arsenic compounds specifically, but
this is exactly the kind of convention question that benefits from human
sign-off rather than silent LLM extrapolation).

**No systematic errors observed.** The model did not hallucinate CAS numbers
for well-known elements/compounds, did not conflate distinct relationship
types, and applied the four-way rule (salt/oxide → parent; already-parent →
direct; mixture → unresolvable; uncertain → flag) consistently across all 30
rows with no `UNCERTAIN — needs human review` outputs in this batch (every
row had a confident classification, which is plausible given the batch was
hand-selected for clarity rather than randomly sampled — the real 587-row run
will likely surface genuinely ambiguous cases this pilot didn't test).

## Recommendation: batch size and full-run estimate

**Batch size: 30–40 chemicals per call.** This keeps each request well within
typical context/output limits, produces a manageable single-call JSON payload
to validate, and gives natural checkpointing — if a batch fails to parse or
the script crashes partway through, at most one batch of work is lost rather
than the whole run.

**Estimated calls for the full 587-row population:** 587 ÷ 35 ≈ **17 calls**
(using a batch size of ~35 as a middle estimate; adjust based on the actual
timing/output-token usage observed on the first live run of
`scripts/stage2b-cas-llm-pilot.R`).

**Estimated runtime:** at an estimated 20–60 seconds per call (unverified —
see scope note), 17 sequential calls would take roughly **6–17 minutes**
wall-clock if run one after another with no parallelism, or a few minutes if
batches are run concurrently (a handful of parallel requests would be safe
given the request volume — Anthropic's standard rate limits comfortably
accommodate &lt;20 requests for a one-off batch job).

**Estimated cost:** Claude Sonnet 4.6 pricing is **$3.00 per million input
tokens / $15.00 per million output tokens** (confirmed current pricing, not
estimated from memory). This pilot's single call used roughly 1,000 input
tokens (system prompt + 30-chemical JSON + instructions) and an estimated
2,500–3,500 output tokens (30 structured JSON objects with rationale text).
Scaling: 17 calls × (~1,000 input + ~3,000 output average) ≈ 17,000 input +
51,000 output tokens total ≈ **$0.0005 + $0.0008 ≈ under $0.01** for the
entire 587-row run at standard pricing. Using the **Message Batches API**
(50% discount, processes asynchronously within ~1 hour) would roughly halve
this — but for a one-off ~17-call job, the latency cost of batch processing
(minutes to ~1 hour turnaround) is unlikely to be worth the trivial cost
saving versus just running synchronous calls directly.

**Bottom line:** cost is a non-issue at this scale (the entire 587-row run is
expected to cost well under $0.01–0.02 in API charges, even with generous
token estimates). The practical considerations for the full run are (1)
verifying the JSON parse-success rate empirically once a real key is
available, since this pilot's "100%" figure is not measured; (2) deciding how
much human review time to allocate, since even a 90%+ "high confidence" rate
across 587 rows still leaves dozens of rows needing a careful look,
especially around convention questions like the Arsenite-vs-Arsenic case
above; and (3) whether to randomly sample a portion of the real 587 for an
additional validation pass before trusting the full batch, since this
pilot's hand-picked 30 chemicals were chosen for clarity and may not surface
the full range of ambiguity in the real population (very long IUPAC names,
unusual salts not matching common patterns, chemicals with genuinely
contested taxonomic parents).
